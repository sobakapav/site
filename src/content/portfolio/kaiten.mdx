## <a name="anchor1" />Бизнес-задача

Увеличить конверсию в регистрацию людей в продукт, оставление заявки.

## <a name="anchor2" />Дизайн-задачи

1. Разобраться, какие информационные ожидания есть у разных аудиторий;
2. Выявить критические точки потери пользователей;
3. Спроектировать UX-решения, которые можно проверить через A/B-тестирование;
4. Подготовить набор wireframe-макетов для внедрения командой клиента.

## <a name="anchor3" />Детали процесса
### <a name="anchor4" />Аналитика

После погружения в нюансы продукта и отраслевые особенности (а у команды был опыт работы с похожими системами: [Elma](/portfolio#elma), [Humans](/portfolio/humans), Кайтен) мы сразу пошли общаться с пользователями. Нам нужно было понять, какие боли есть у аналитиков и руководителей бизнеса, чтобы новый модуль отвечал на нужные им вопросы. 

<a href="/images/portfolio/storm/4.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/4.png)</a>
_Отчет. Сценарии оптимизации_

Обсудив с заказчиком их текущие гипотезы, мы выделили те группы пользователей, которые нам подходят: аналитики среднего и крупного бизнеса, которые строят массовые процессы (HR, тендеры, производство). Это могут быть как люди, работающие ближе к бизнесу, так и те, кто занимается в первую очередь анализом big data для оптимизации. 

<a href="/images/portfolio/storm/5.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/5.png)</a>
_Отчет. Экспертный анализ_

В итоге поговорили с 8 пользователями. Их расспрашивали про инструменты и методы, которые они используют для анализа процессов, и про место,  которое StormBMPN занимает среди этих инструментов занимает. Информанты рассказывали про имитационное/математическое моделирование процессов, симуляцию прохождения заявок, моделирование нагрузки и поиск узких мест по времени, SLA, ресурсам и стоимости.

Закодировали результаты и собрали код-бук, который прямо в рабочем виде отдали заказчику: он отлично знал, что с ним делать дальше. Так же сделали профилирование пользователей, которое показало еще одну глобальную цель продукта: тащить пользователей из левого верхнего и правого нижнего квадрантов наверх вправо. То есть наметился образовательный слой будущего интерфейса. 

<a href="/images/portfolio/storm/1.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/1.png)</a>

Отдельно восхищаемся комьюнити вокруг продукта: пользователей для качественного исследования мы набрали за 30 минут. Заказчик просто написал в канал и нам пришлось не просто экстренно закрывать объявление, но и отдать невместившихся внутренней команде на разговор по нашему гайду. Такого на нашей памяти еще не бывало.

<a href="/images/portfolio/storm/6.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/6.png)</a>
_Отчет. Интерактивная BPMN диаграмма_

## <a name="anchor4" />Дизайн

Как часто бывает в проектировании, сам интерфейс – это только часть жизненной ситуации пользователя. Она не возникает в момент работы в интерфейсе, наоборот – ты приходишь в интерфейс решить уже сформулированную задачу. И после того, как результат получен, часто его использование происходит уже вне этого ПО (да и вне компьютера тоже).

<a href="/images/portfolio/storm/7.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/7.png)</a>
_Создание симуляции_

Здесь было именно так. Бизнес-процесс рождается задолго до продукта, и результаты моделирования обсуждаются на совещаниях со слайдами и распечатками. Но для понимания всей картины мы все-таки собрали некое подобие CJM, куда включили весь путь аналитика от осознания задачи до её завершения. И на обсуждении с заказчиком очертили зону ответственности нового функционала на этом пути.

<a href="/images/portfolio/storm/8.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/8.png)</a>
_Процесс симуляции_

Дальше еще один новый, но уже вошедший в наш инструментарий этап: концептуальное проектирование с помощью ИИ. На этом проекте мы использовали Figma Make и прототип заказчика, который они собирали тоже с помощью нейросети. Мы сделали вот что: взяли текущие макеты прототипа, дополнили наш базовый промт для такой задачи (по умолчанию Make делает слишком много отсебятины) проблемами этого интерфейса, найденными по итогу аналитики. 

<a href="/images/portfolio/storm/9.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/9.png)</a>
_Результаты в правой панели_

После череды повторений и уточнений первый прототип был готов для обсуждения с заказчиком. В течении пары дней мы поштормили идеи, поговорили с заказчиком и сформулировали, что берем в проработку.

<a href="/images/portfolio/storm/10.png" target="_blank">![Собака Павлова • Storm BPMN • Система для построения бизнес-процессов](~/assets/images/portfolio/storm/10.png)</a>
_Формирование отчета_

Дальше все очень стандартно – три итерации проектирования отдельных частей интерфейса (симулятор, формирование отчета и сам отчет). Потом общая  вычитка макетов и внесение мелких правок.
